# Code Review: flora-root 内部代码重复问题

**日期**: 2026-07-24
**范围**: `flora-root` 模块（零依赖工具库，380 源文件）
**目的**: 排查同一能力在多个包中重复实现的问题（本可直接调用 root 内其他包）
**规则**: 只排查，不修改

---

## 1. 十六进制编解码 —— 三重实现（严重）

### 涉及的类

| 类 | 方法 | 可见性 |
|---|---|---|
| `com.flora.codec.HexUtil` | `encodeHex(byte[])`, `decodeHex(String)`, `isValidHex(String)` | public，公开 API |
| `com.flora.data.BytesUtil` | `bytes2hexString(byte[])`, `hexString2bytes(String)` | public |
| `com.flora.entropy.hash.StandardHash` | `bytesToHex(byte[])` | private |

### 细节

**`BytesUtil.java`** (`flora-root/src/main/java/com/flora/data/BytesUtil.java`):

```java
// ≈ line 30-40
private static final char[] HEX_ARRAY = "0123456789ABCDEF".toCharArray();

public static String bytes2hexString(byte[] bytes) {  // 与 HexUtil.encodeHex 语义相同
    char[] hexChars = new char[bytes.length * 2];
    for (int i = 0; i < bytes.length; i++) {
        int v = bytes[i] & 0xFF;
        hexChars[i * 2] = HEX_ARRAY[v >>> 4];
        hexChars[i * 2 + 1] = HEX_ARRAY[v & 0x0F];
    }
    return new String(hexChars);
}

public static byte[] hexString2bytes(String s) {  // 与 HexUtil.decodeHex 语义相同
    // ... 独立实现，异常类型为 InvalidParameterException
}
```

**`StandardHash.java`** (`flora-root/src/main/java/com/flora/entropy/hash/StandardHash.java`):

```java
// ≈ line 320
private static final char[] HEX_CHARS = "0123456789abcdef".toCharArray();

private static String bytesToHex(byte[] bytes) {  // 与 HexUtil.encodeHex 完全重复
    char[] hexChars = new char[bytes.length * 2];
    for (int j = 0; j < bytes.length; j++) {
        int v = bytes[j] & 0xFF;
        hexChars[j * 2] = HEX_CHARS[v >>> 4];
        hexChars[j * 2 + 1] = HEX_CHARS[v & 0x0F];
    }
    return new String(hexChars);
}
```

### 分析

三个类各有一套独立的 hex 编解码：`HexUtil` 是公开编解码 API，`BytesUtil` 在其工具集中重新实现了一遍（使用大写字母），`StandardHash` 在内部私有方法中又实现了一遍（使用小写字母）。核心算法完全一致——通过位移查表转换。

**起点不同**：`HexUtil.encodeHex` 输出小写，`BytesUtil.bytes2hexString` 输出大写，`StandardHash.bytesToHex` 输出小写。但小写转大写只是一个 `toUpperCase()` 调用。

**修复建议**：`BytesUtil.bytes2hexString` 改为委托 `HexUtil.encodeHex` 后 `toUpperCase()`；`StandardHash.bytesToHex` 改为直接调用 `HexUtil.encodeHex`。

## 2. 字符串判空与默认值 —— 工具类间重复

### 涉及的类

| 类 | 方法 | 说明 |
|---|---|---|
| `com.flora.data.StrUtil` | `isBlank(CharSequence)`, `isEmpty(CharSequence)`, `defaultIfNull(String, String)` | 字符串专用 |
| `com.flora.container.ObjectUtil` | `defaultIfNull(T, T)` | 泛型版本 |
| `com.flora.java.CheckUtil` | `notNull(T, String)`, `notEmpty(CharSequence, String)` | 带异常抛出的校验 |

### 分析

`ObjectUtil.defaultIfNull(obj, def)` 可以处理 String，与 `StrUtil.defaultIfNull(str, def)` 语义完全重叠。`CheckUtil` 的 `notEmpty(CharSequence)` 与 `StrUtil.isNotEmpty` 是同一件事的两个方向。

**修复建议**：`StrUtil.defaultIfNull` 可直接委托 `ObjectUtil.defaultIfNull`；`CheckUtil` 相关的判空校验可委托 `StrUtil.isEmpty/isBlank`。

## 3. 集合判空与操作 —— CollectionUtil 与 IterableUtil 重叠

### 涉及的类

| 类 | 方法 |
|---|---|
| `com.flora.container.CollectionUtil` | `isEmpty(Collection)`, `isEmpty(Map)`, `filter`, `map`, `first` |
| `com.flora.data.IterableUtil` | `isEmpty(Iterable)`, `filter`, `map`, `first` |

### 分析

`Collection` 是 `Iterable` 的子接口。`CollectionUtil.isEmpty(Collection)` 和 `IterableUtil.isEmpty(Iterable)` 对 Collection 输入做同样的事。同理 `filter`、`map`、`first` 两组方法语义重叠。

`CollectionUtil` 操作 `Collection<E>` 返回 `List<E>`；`IterableUtil` 操作 `Iterable<E>` 返回 `List<E>`。当输入是 Collection 时，两者的行为一样。

**修复建议**：将 `IterableUtil` 中对 `Collection` 的逻辑委托给 `CollectionUtil`，或反过来。至少确保二者不各自维护一套独立的算法。

## 4. 哈希摘要内的私有 hex 方法 —— StandardHash

**已在 §1 中覆盖**。`StandardHash.bytesToHex` 是私有的，但这是最容易修复的——直接删掉私有方法，改为调用 `HexUtil.encodeHex`。

---

## 5. MemoryCache 内部 Count-Min Sketch

`com.flora.cache.store.MemoryCache` 内部类 `FrequencySketch` 实现了 4-bit 饱和计数的 Count-Min Sketch，而 `com.flora.entropy.probds.CountMinSketch` 是独立的通用实现。

**严重程度：低**。`FrequencySketch` 是 4-bit 饱和计数（专为 TinyLFU 优化，内存密集），通用版本是 long 矩阵。两者设计目的不同，但算法本质相同。如果要消除重复，可以将 `FrequencySketch` 改为包装通用 `CountMinSketch` 的 4-bit 适配层。

---

## 6. 两套 Tuple 系统

`com.flora.container.tuple.Tuple1~20`（泛型 Object[]）与 `com.flora.fast.container.tuple` 的 FastTuple（原始类型特化，代码生成产物）都提供多值容器语义。

**严重程度：低**。原始类型特化是为了避免装箱开销，属于性能优化。但使用时需要清楚区分两套 API，增加了认知负担。

---

## 7. BytesUtil 与 ByteArrayBuilder 的字节数组转换 —— 字节序不同

`com.flora.data.BytesUtil` 提供了 `short2bytes`/`int2bytes`/`long2bytes` 等 **小端序** 转换，`com.flora.classfile.ByteArrayBuilder` 提供了 `writeShort`/`writeInt`/`writeLong` 等 **大端序** 转换。

核心算法相同，只是字节顺序反转。

**严重程度：中**。如果 `ByteArrayBuilder` 可以委托 `BytesUtil` 后做字节反转，或将 `BytesUtil` 扩展为支持指定字节序，可以减少一份实现。

## 总结

| 编号 | 问题 | 位置 | 严重程度 |
|------|------|------|----------|
| 1 | **Hex 编解码三重实现** | `HexUtil` / `BytesUtil` / `StandardHash` | 严重 |
| 2 | **判空与默认值重复** | `StrUtil` / `ObjectUtil` / `CheckUtil` | 中 |
| 3 | **集合操作重复** | `CollectionUtil` / `IterableUtil` | 中 |
| 4 | StandardHash 私有 hex 方法 | `StandardHash.bytesToHex` | 轻（已含在 #1） |
| 5 | 内部 Count-Min Sketch | `MemoryCache.FrequencySketch` | 轻 |
| 6 | 两套 Tuple | `container.tuple` / `fast.container.tuple` | 轻（设计取舍） |
| 7 | 字节数组转换（字节序） | `BytesUtil` / `ByteArrayBuilder` | 中 |

**最优先修复**的是 #1（hex 编解码三重实现），因为它涉及三个类，且修复方式简单直接——**`BytesUtil.bytes2hexString` 和 `StandardHash.bytesToHex` 都应当委托 `HexUtil.encodeHex`**，消除重复代码。
