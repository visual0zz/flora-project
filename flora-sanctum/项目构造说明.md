# Flora Sanctum — 密码管理工具 项目构造说明（任务书）

> 本文档是面向其他 Agent 的任务书（spec / 构造说明），描述一个**低依赖、现代 GUI、基于 JGit 同步**的
> 本地优先（local-first）密码管理工具的设计与实现要求。后续 Agent 应以此文档为准进行开发，若发现
> 文档与实现存在冲突，以“尽量低依赖 + 功能完整 + 可运行”为最终裁决原则，并回写本文件记录偏离理由。

---

## 1. 项目目标

构建一款名为 **Flora Sanctum**（圣所，寓意存放秘密之处）的密码管理工具：

- **核心能力**：安全存储账号/密码/备注等凭据，使用主密码加密，本地优先。
- **同步能力**：以 Git 仓库作为数据载体，借助 **JGit** 完成提交、拉取、推送，把加密后的保险库
  （vault）同步到任意 Git 代码托管平台（GitHub / GitLab / Gitee / 自建 Git 服务等）。
- **界面能力**：提供**现代化 GUI**，交互流畅、外观整洁（暗色主题、搜索、分类）。
- **依赖约束**：**尽量低依赖**。除 GUI 框架 **JavaFX**（无法以源码形式 vendor，且为平台模块）外，
  所有第三方能力——**JGit 同步**、**BouncyCastle 的 Argon2**——均以**源码 vendor（复制进工程）**方式纳入，
  **不挂任何 Maven 依赖**；其余优先使用 JDK 内置能力。

## 2. 技术选型与依赖约束（关键决策）

### 2.1 语言与构建

- **Java 26**，作为本仓库（`flora-project`）下的一个 Maven 子模块，纳入 JPMS 模块化体系。
- 模块名：`com.flora.sanctum`。
- 加入根 `pom.xml` 的 `<modules>`；新建 `flora-sanctum/pom.xml`，继承根 POM 的 Java 版本与编码约定。

### 2.2 GUI 框架（唯一的大依赖）

- 使用 **JavaFX**（模块化引入 `javafx.controls` / `javafx.fxml` / `javafx.graphics`）。
- 选择理由：JDK 自带的 Swing 虽零依赖但外观陈旧、开发体验差；“现代 GUI”目标下 JavaFX 是
  **最低成本且最现代**的官方方案。仅引入上述必要模块，不引入 media / web 等无关模块。
- 通过 `javafx-maven-plugin` 或 JDK 的 `jlink`/`jpackage` 完成运行与打包；任务中需给出可双击启动的方式。

### 2.3 同步层（JGit 源码 vendor，不挂 Maven 依赖）

- 使用 **JGit**（`org.eclipse.jgit`，纯 Java 实现，无需本地安装 Git）完成仓库的
  init / clone / commit / push / pull。
- **硬性要求：将 JGit 源码直接复制（vendor）进 `flora-sanctum/src/main/java`，不得通过 Maven 依赖引入。**
  与“尽量低依赖”一致，使整个模块自包含、运行期不依赖外部 jar。
- **方法**：克隆 JGit 仓库到**临时目录**（`absent/tmp`，已被 `.gitignore` 忽略），复制所需包到工程、
  保留原始包名 `org.eclipse.jgit...`，递归补齐其依赖（核心是 `com.googlecode.javaewah` 的 bitmap
  实现），用后删除临时克隆。具体步骤见 §2.3.1 Agent 提示词。
- **传输方式**：默认支持 `file://`（本地裸仓测试）与 `https`；若需 SSH 推送，则把
  `Apache MINA sshd` 相关类也一并 vendor（或暂不支持 SSH 并在 §10 登记选择）。避免引入
  `jgit.archive` 等无关模块。
- **许可**：JGit 为 EDL-1.0（等同 BSD-2-Clause），vendor 时**必须保留每个文件的版权/许可声明**（见 §11）。

### 2.4 加密（全部使用 JDK 内置，零额外依赖）

- 加密算法：**AES-256-GCM**（JCE 内置，`javax.crypto`）。
- 密钥派生：主密码 → **PBKDF2-HMAC-SHA256**，高迭代次数（≥ 600,000，可按设备性能调参），
  派生出 256-bit 密钥 + 随机 96-bit IV/nonce。
- 完整性：GCM 自带认证标签，无需额外 MAC。
- 随机数：`java.security.SecureRandom`。
- 默认 KDF 为上述 PBKDF2（JDK 内置）；如需更强抗暴力破解的 **Argon2（memory-hard）**，按 §2.7 将
  BouncyCastle 的纯 Java 实现以**源码 vendor**方式引入（同样不挂 Maven 依赖）。

### 2.5 数据序列化（复用 flora-root，零额外依赖）

- JSON 能力**直接依赖本仓库的 `flora-root` 模块**（零依赖工具库），复用其 `com.flora.codec` 包下的
  `JsonUtil` / `JsonParser` / `JsonBuilder`（以及 `JsonPath` / `JsonIgnore` 等），不手写、也不引入
  Jackson/Gson 等第三方 JSON 库。
- `flora-sanctum` 的 `pom.xml` 与 `module-info.java` 需 `requires com.flora.root;`（flora-root 已导出
  `com.flora.codec`）。条目模型（`Entry` / `VaultMeta`）可用 `@JsonIgnore` 等注解配合 `JsonUtil` 序列化。
- 二进制加密载荷以 Base64（JDK `java.util.Base64`）存入文本文件，便于 Git 文本 diff 与合并。

### 2.7 加密增强：Vendor BouncyCastle 纯 Java Argon2 源码（零 Maven 依赖）

**动机**：第 2.4 节的 PBKDF2 仅消耗 CPU（可并行暴力破解）；Argon2 是**内存困难（memory-hard）**的
KDF，抗 GPU/ASIC 更强。但 JDK 不内置 Argon2，而“尽量低依赖”要求不挂 Maven 依赖。方案：**把
BouncyCastle 的纯 Java Argon2 实现直接 vendor（复制源码）进本模块**，从而既用上 Argon2，又不增加
任何 Maven 依赖项（与 §2.3 的 JGit vendor 思路一致）。

**可行性**：BC 的 `Argon2BytesGenerator` 是纯 Java（无 JNI），Spring Security 的 `Argon2PasswordEncoder`
底层即它。主流 `argon2-jvm` 等是 JNI/JNA 调 C 库的壳，**不可用于此方案**（需原生二进制、不跨平台）。

**方法（核心思路）**：先把 BouncyCastle 仓库 clone 到一个**临时文件夹**（本项目 `absent/tmp` 已被
`.gitignore` 忽略，不会进版本库），从中只挑出 Argon2 相关类复制进 `flora-sanctum/src/main/java`，
保留原始包名；临时克隆仅供审查，用后删除。

**需复制的最小文件集**（已核实依赖链）：

- `org.bouncycastle.crypto.generators.Argon2BytesGenerator`
- `org.bouncycastle.crypto.params.Argon2Parameters`
- `org.bouncycastle.crypto.digests.Blake2bDigest`（Argon2 内部哈希依赖）
- 支撑类（递归补齐）：`Digest` / `ExtendedDigest` / `CharToByteConverter` / `PasswordConverter` /
  `CryptoServicePurpose` / `CryptoServicesRegistrar` / `OutputLengthException`，以及
  `util` 下的 `Arrays` / `Longs` / `Pack` / `Properties` 等。

**许可**：BC 为 Bouncy Castle / MIT 双许可，vendor 时**必须保留每个文件原有的许可证头**，并在
第 10 节偏离登记中注明来源（仓库 + 分支/commit）与文件清单。

> 本方案为**推荐增强**：用上 memory-hard KDF 且仍零 Maven 依赖；执行后须在 §10 / §11 登记来源与许可。

## 4. 数据模型

```text
VaultMeta:
  formatVersion: int          # 格式版本，便于升级
  kdf: "PBKDF2-HMAC-SHA256"
  iterations: int             # 派生迭代次数
  salt: base64                # 随机盐，防彩虹表
  remotes: List<RemoteInfo>   # 远程仓库信息（明文区，见下；仅非机密元数据）
  createdAt / updatedAt: long

RemoteInfo:                    # 明文存放，不含任何凭据明文
  url: string                  # 远程仓库地址（HTTPS/SSH）
  protocol: "HTTPS" | "SSH"
  username: string             # 非机密用户名/标识
  # 注：SSH 私钥、token 等敏感凭据不在此处，作为加密 Entry 存于 entries/ 区

Entry:
  id: uuid                    # 稳定标识，便于合并
  title: string               # 显示名
  username: string
  password: string
  url: string
  category: string            # 分类/文件夹
  note: string
  createdAt / updatedAt: long

Vault:
  meta: VaultMeta
  entries: List<Entry>        # 加密后以单文件或分文件存储
```

- **存储策略（默认，松散文件）**：保险库本身就是一个**文件夹**，该文件夹同时是 Git 仓库。
  采用“一堆松散文件”的方式，便于 Git 按条目粒度 diff / 合并，冲突影响面最小：
  - `meta.json`（**明文区**）：明文存放 `VaultMeta`，含 `formatVersion`、`kdf`、`iterations`、
    `salt`、时间戳，以及**远程仓库信息 `remotes`**（`url`、`protocol`、`username` 等非机密元数据）。
    salt / iterations / 远程 URL 等不属于机密，可明文；**明文区绝不存放任何凭据明文**（SSH 私钥、
    token 等绝不出现在此）。
  - **明文区防篡改（哈希签名）**：对 `meta.json` 明文内容计算 `HMAC-SHA256`（密钥由主密码派生，
    与加密密钥同源或派生独立 MAC 密钥），签名值随仓库保存（如 `meta.json.sig`）。解锁时先校验签名，
    若不匹配则拒绝加载并明确提示“明文数据可能被篡改”（如远程 URL 被恶意改写指向钓鱼仓库）。
  - `entries/<uuid>.enc`：每条 `Entry` 单独一个加密文件。内容为 AES-GCM 加密后的该条目 JSON，
    再以 Base64 文本落盘。文件名使用条目的 `uuid`，避免改名导致 Git 重命名抖动。
    **SSH 私钥、访问 token 等敏感凭据以普通 `Entry` 形式加密存于此区**，与其他密码条目一视同仁
    （例如一条 `title=“GitHub SSH 私钥”` 的加密条目），不单独走明文或外部配置。
  - **不维护集中式索引文件**：文件夹列表即索引；解锁时遍历 `entries/` 解密全部条目到内存，
    保存时仅重写被修改的那一个文件 → Git 只产生最小 diff，天然支持按条目合并。
  - 新增条目 = 新增一个 `.enc` 文件；删除条目 = 删除对应文件；改条目 = 重写对应文件。
- **不使用单文件大 blob**：放弃“整个 Vault 加密为单个 `vault.enc`”的方案，因其任一字段变动都会
  让整个文件 diff 失效、且冲突即全库冲突。松散文件是本项目为 Git 友好而做的核心设计取舍。

## 5. 加密方案（强制要求）

1. 首次创建保险库：生成随机 `salt`（≥16 字节）与 `iterations`，存入 `VaultMeta`（明文，无密钥信息泄露）。
2. 主密码 + salt + iterations → `PBKDF2-HMAC-SHA256` → 32 字节密钥。
3. 每次加密生成新随机 12 字节 IV；输出 `IV || ciphertext || tag`，Base64 后写入文件。
4. 解密失败（密码错误 / 数据损坏）→ 明确提示，不崩溃。
5. **绝不在内存/磁盘明文落盘主密码**；密钥仅在解锁期间驻留，锁屏/退出时清零（`Arrays.fill`）。
6. 提供“修改主密码”能力：重新派生并重新加密整个 vault，并同步更新明文区签名。
7. **明文区完整性（防篡改）**：由主密码派生独立 MAC 密钥（或对同一派生密钥计算 HMAC），对明文存储区
   （`meta.json`，含远程仓库信息）计算 `HMAC-SHA256` 签名并存于仓库。每次解锁先校验签名：若不匹配
   （如远程 URL 被恶意改写指向钓鱼仓库），拒绝加载并明确提示“明文数据可能被篡改”。

## 6. JGit 同步机制

- **本地仓库**：保险库目录本身即一个 Git 仓库（由 `GitBackend.init()` 创建，或 `clone()` 已有远程）。
- **提交**：每次保存后自动 `add` + `commit`（message 含时间戳，不含明文凭据）。
- **远程**：用户在 `SyncSettings` 中配置远程仓库。**远程 URL、协议类型（HTTPS/SSH）、用户名等非机密
  元数据明文存入 vault 明文区（`meta.json`）**；而 **SSH 私钥、访问 token 等敏感凭据以普通加密
  `Entry` 形式存于 `entries/` 加密区**，与其他密码条目一视同仁，绝不写入明文区。明文区受哈希签名
  保护（见 §4 / §5），若远程 URL 等被篡改，解锁时签名校验将失败并告警。
- **推送**：手动或定时 `push` 到 `origin`。支持多平台（GitHub/GitLab/Gitee）。
- **拉取与冲突（按文件）**：推送前先 `pull`。由于采用松散文件，Git 的冲突被限制在
  **被两端同时修改的那一个条目文件**上，其余条目完全不受影响：
  - 同一 `.enc` 文件被并发修改时，Base64 密文无法被 Git 文本合并 → 产生冲突标记；
  - 默认策略：**保留本地最新**（last-write-wins）并提示用户哪条条目发生了冲突；
  - 进阶：冲突时保留双份（如 `<uuid>.enc.local` 与 `<uuid>.enc.remote`）供用户在 GUI 中二选一。
  - 不实现密文级三方合并算法；以“条目文件”为最小冲突单位。
- **离线优先**：无网络时本地照常工作，联网后再同步；同步失败不影响本地使用。

## 7. GUI 设计要求（现代）

- **解锁页**：主密码输入框（遮罩）、创建/解锁切换、错误提示、可选“显示密码”。
